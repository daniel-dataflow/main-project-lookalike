# 시스템 코어 트러블슈팅 안내 가이드 명세서

## 1. FastAPI 언어 이벤트 루프 블로킹(Event Loop Blocking) 병목 이슈 및 자원 지연 개선 건

### 1.1 시스템 문제 상황 (Issue Report)
- **증상 스펙**: 어드민 데스크 페이지 및 사용자 메인 웹 페이지 초기 로딩 통신 시, 간헐적으로 **응답률 최대 20초 스팬에 달하는 극심한 로딩 지연(Timeout Hang)** 마비 현상이 드랍됨.
- **영향도 타격**: 메인 시스템 서버 엔진의 응답 병목 병세로 정상적인 클라이언트 서비스 이용이 원천 불가함. 사용자 경험(UX) 품질에 치명적인 접속 장애 폴트를 유발함.

### 1.2 장애 원인 집중 분석 (Root Cause Analysis 진단)
사태의 근본 발원지 원인은 코어 자원 **FastAPI의 메인 이벤트 루프(Event Loop) 타스크 락킹(Blocking)** 간섭 현상으로 판명 진단됨.

FastAPI 아키텍처는 파이썬 코어의 `asyncio` 비동기 코루틴을 스케줄링하여 싱글 스레드 런타임 기반의 이벤트 루프에서 수많은 입출력 요청을 비동기적으로 동시에 레이어링 처리함. 즉, 모든 웹 페이지 진입 렌더링 요청과 외부 API 데이터 통신 I/O가 단일 메인 워커 스레드를 교차 컨텍스트 스위칭하여 공용 이용함. 그러나 신규 개발된 백그라운드 스케줄러 컨트롤러 코드(`log_collector.py`, `metric_collector.py`) 모듈 내부에 타임 컨슈밍이 긴 무거운 인프라 처리 동기형 로직이 탑재된 점이 치명적인 병목 락(Lock) 오류를 파생시킴.

```python
# [과거 디버그 검출 대상] 장애 발생 유발 기존 코드 덩어리
async def start_background_collection(self):
    while True:
        self.collect_metrics()  # 해당 함수의 동기 블로킹 발생점
        await self.run_collection_cycle() 
        await asyncio.sleep(15)
```
- 지목된 수집 모듈 컴포넌트 하위 내부에는 Docker SDK 소켓을 직접 통한 각 서브 컨테이너 헬스 상태 집계 연산, 외부 파이프라인 API 긴 통신 등 I/O 단일 처리 스팬이 매우 긴 **동기적(Synchronous) 타스크 작업**들이 무분별하게 포함 코딩되어 있음.
- 결과적으로 이 I/O 바운드 무거운 작업들이 메인 루프 스레드 위에서 그대로 직접 선탑재 실행되며, **로직 수행이 자체 종료 완료될 때까지 평균 20초 이상 코어 이벤트 루프의 연산 제어권을 독점 락킹(Blocking)해 버림.**
- 결과론적 여파로 도커 백그라운드 데이터 수집 데몬이 상태값을 긁어오는 연산 스팬 동안, 메인 시스템 서버는 신규 클라이언트 접속과 HTML 페이지 렌더링 소스를 일절 응답하지 못하고 모든 자원이 멈추는 `일시 정지(Hang)` 병목 셧다운 상태에 지속 직면함.

### 1.3 소프트웨어 엔지니어링 패치 해결 방안 (Technical Solution)
장시간 무거운 자원 연산을 연속 유발하는 치명적인 동기적 로직 타스크 코드들을 코어의 메인 이벤트 루프와 강력히 샌드박스로 격리 배제하기 위해 `asyncio.to_thread` 모듈을 소스 코드 레벨 전체에 전면 개편 주입 도입함.

```python
# [구조 리팩토링 및 픽스] 메모리 자원 개선이 반영된 신규 안전 적용 코드
async def start_background_collection(self):
    while True:
        # 오버헤드가 강한 동기 작업을 FastAPI 메인 컨텍스트에서 오프로딩(Offload) 배제하여 파이썬 내장 별도의 워커 스레드 인스턴스로 분리 멀티 분산 이양함.
        await asyncio.to_thread(self.collect_metrics)
        await asyncio.to_thread(self.run_collection_cycle)
        await asyncio.sleep(15)
```
- **빌트인 `asyncio.to_thread()` 구조**: 타임라인 블로킹이 상존하는 무거운 함수 객체를 인자로 배정 매핑할 시, 이를 현재 진행 중인 메인 스레드가 아닌 파이썬 인터프리터 내부 백그라운드망의 **별도 커스텀 스레드 풀(Thread pool)** 자원에 안전하게 권한 이양하여 병렬 실행되도록 지원 스위칭함.
- 로직 재배치 전환을 통해 어드민 인프라 상태 수집 집계에 딜레이가 수십 초가 소모 지연되더라도, 메인 웹서버 이벤트 루프는 워커 풀에 임무 위임 즉시 제어 사이클 통제권을 탈환 획득하여 **추가로 유입된 타 웹 핑 접속이나 무수한 API 통신 요청들을 큐 점유 딜레이 없이 쾌속 병렬 처리하도록 보장됨.**

### 1.4 인프라 구동 효과 지표 (Impact Output)
기존 모놀리식 강제 함수를 단편적인 스레드 오프로딩(Offloading) 처리를 이양 거침을 통해 다음과 같은 유의미한 시스템 성능 개선 척도가 확인 산출됨.
1. **메인 이벤트 루프의 큐 체증 병목 완벽 통제 해소**: 스레드 풀 격리 분리 설계로 단말 로딩 20초 지연 멈춤 타임아웃 셧다운 현상이 100% 완전 해소 근절됨.
2. **서버 API 동시성 처리량(Throughput) 대역 파이프라인 정상화 방어**: 자원 병목 데드락 사태가 타파되어 쾌적한 렌더링 페이지 응답 속도 반환이 확보 및 FastAPI 비동기 본연의 벤치마크 퍼포먼스 수치로 복구 증명됨.

## 2. 관리자(Admin) 전용 페이지 인가 403 Forbidden 권한 무반응 에러 폴트 건

### 2.1 시스템 문제 상황 모니터링
- **증상 현상**: 런타임 환경 변수 `.env` 설정 셸에 정상 기록된 올바른 `ADMIN_PASSWORD` 슈퍼 키값을 보안 UI 폼에 기입하고도 시스템 통신망 상 `[ 시스템 관리자 접근 권한이 부족합니다 (403 Error 리턴) ]` 인증 거절 경고가 무분별 노출되며 타겟 게시판 진입 라우팅이 원천 거부되는 장애.
- **근본 원인 파악**: 초기 프로젝트 시스템 오픈 당시 구 스키마 DB 내 `users.role` 값이 `'ADMIN'` 배열 문자열인지 쿼리를 통해 직접 검색 대조하는 어설픈 하드코딩 정합성 락(Lock) 로직이 레거시에 그대로 포함되어 있었으나, 실제 라이브 데이터베이스 도메인 내엔 초기 값인 전원 일반 `'USER'` 권한뿐이기에 정당한 텍스트 비밀번호가 인입 입력되어도 쿼리 결과에 따라 롤오버 에러를 리턴해 튕겨버리는 치명적 아키텍처 결함 코드 버그.

### 2.2 해결 방안 조치 내역 (Fix)
- **코드 패치 배포**: 관리자 접근 권한 체계를 기존 DB 계정 RDBMS의 역할 권한 컬럼 조회를 통해 분류하는 1차원 하드코딩 모델망을 전면 폐기 철수함. 그 대신, **순수 Redis 인메모리 브라우저 세션 의존 플래그** 뱃지 방식 부여 설계로 분리 도입을 아키텍처에 강제 적용 반영함.
- 신규 폼 패스워드 입력 인증 모듈망이 올바른 패스워드로 검증 연산 통과 달성 시, 런타임 메모리 백엔드의 접속 중 활성 세션(Session dictionary payload) 노드 내에 `is_admin: true` 불리언(Boolean) 권한 인덱스를 단일 추가 주입하는 비동기 이벤트를 발생시킴. 
- 이후, 시스템 백본 깊은 뎁스 라우터에서 뿜어지는 기존 API 보안 미들웨어 검증용 `_require_admin()` 함수가 더 이상 DB를 찾지 않고, 핑 세션의 해당 뱃지 키페어 도출 보유 여부만 독립 판단하게 개선 리팩토링 파이프라인하여, 패스워드 검열 인증 실패 난제 폴트를 시스템적으로 완벽 통제 해소 격리함.

## 3. 웹 소셜 오픈 로그인 API 연동 간 보안 모듈 Passlib & Bcrypt 버전 비호환 충돌 붕괴 사태 (HTTP 500 내부 에러)

### 3.1 시스템 문제 상황
- **증상 현상**: 신규 소셜 회원가입/로그인 인증 플로우 통신을 연동하는 트랜잭션 도중, API 메인 서버 엔진 런타임이 무한 멈추며 브라우저 시스템 500 디버그 에러 컨테이너 패널 송출 및 `ValueError: passlib.hash...` 치명적 백엔드 파이썬 예외 처리(Exception) 로깅 로그 발생 출력.
- **원인 파악**: FastAPI 프레임워크 표준 튜토리얼 문서 등에서 범용 보안 의존성 패키지로 흔히 인스톨 채택하는 구식 `passlib` 패키지 모듈이, 2024년 이후 대대적으로 업그레이드 개편된 최신 파이썬 코어의 `bcrypt 5.x` 리비전 패키징 빌드들과 코어 C언어 빌딩 단에서 메모리 오버플로우 호환 충돌 에러가 속출 지연되는 파이썬 생태계 글로벌 버전 파편화 이슈 여파의 맹점으로 지적판명됨.

### 3.2 패치 해결 방안 (Patch Execution)
- 낡은 아키텍처 구조인 기존 `requirements.txt`에 강제 내재된 언스테이블 `passlib` 의존성을 과감히 배제 삭제 조치 클린 빌드함.
- 불안정한 별도의 외부 서드파티 래퍼 브릿지를 덮어 우회 의존하지 않고, 파이썬의 표준 빌트인 코어 보안 모듈인 **`bcrypt` 암호화 라이브러리를 직접 커스터마이징 스크립트로 호출 연산**하도록 자사 암호화 코어 툴(`utils/security.py`) 시스템 스크립트 라인을 C레벨 전면 재설계 오버라이딩함.
- 덤으로 레거시 구 문법(오래된 SHA256 규격)으로 암호화 구형 저장되어 보안 취약 판정을 받던 초창기 초기 유저 DB 데이터 덤프들을 별도의 1회성 스크립트를 통해 신형 Bcrypt 컨버전 마이그레이션 적용을 수동 루프로 일괄 수행 완료해 백엔드 시스템 스키마 호환성 버전업 락킹(Locking)을 영구 해결 조달함.

## 4. 어드민 UI 렌더링 무한 로딩 지연 (평균 25초 ~ 60초 이상 병목)

### 4.1 시스템 문제 상황 
- **장애 증상**: 사이트 관리자 권한으로 메인 관제 페이지(`/admin/infra` 등) 진입 시도 시 프론트엔드 DOM 로딩이 심각하게 지연되거나, 브라우저 스레드가 장시간 빈 화면으로 강제 대기(Hang)하는 크래시 유발.
- **근본 원인 (Root Cause)**: 
  1. 각 개별 컨테이너 데몬마다 호스트 Docker `stats()` API를 동기식(직렬)으로 무식하게 순차 호출하여 폴링하므로, 전체 인프라 순회에 치명적 네트워크 지연 시간 누적 발생 (13개 컨테이너 × 2초 통신 대기 = 총 26초 스택 소요).
  2. 프론트엔드 네비게이션 진입 라우팅 때마다 권한 인증망을 거치는 방어 API(`/check`, `/me`)를 순차적으로 직렬 호출 타격함에 따른 워터폴 타임 지연 부하 중첩.

### 4.2 패치 해결 방안 (Architecture Optimization 튜닝)
- **백엔드 아키텍처 대공사 변경**: Docker API를 요청마다 매번 동기식으로 직접 찔러 호출하던 구시대적 방식(1세대)을 폐기하고, 백그라운드 인프라 데몬(MetricCollector)이 15초 슬립 단위로 비동기 긁어온 지표를 **Kafka 큐망 → Elasticsearch 스토리지망 (2세대 체제)** 으로 파이프라인화 적재하여 ES DB망에서 단건 즉시 조회(`100ms` 수준 타임컷)로 리턴하도록 초고속 렌더링 재설계 반영함.
- **프론트엔드 병렬 비동기 호출 & 로컬 캐싱망 구축**: 직렬 워터폴 인증 API 통신망을 `Promise.all()` 비동기 병렬 호출 방식으로 구조 최적화하고, 성공된 인증 결괏값을 브라우저 `sessionStorage` 객체 볼륨에 5분 한시 캐싱하여 불필요한 HTTP 왕복 I/O 낭비 부하를 원천 제거함.
- **문서 프리로드(Preload) 기법 적용**: 부모 캔버스 `admin_base.html`에 사이드바 라우팅 링크 hover 터치 이벤트 발현 시 `prefetch` 동적 네트워크 망을 선행 생성 오픈하고, 핵심 데이터 API를 `window.__preload` 전역 변수 트리거로 선행 초기 호출 적용하여 체감 렌더링 속도 부스팅 기법 도입함.

## 5. 하위 Elasticsearch(ES) 불안정에 따른 앱 메인 기동 크래시(Crash Loop)

### 5.1 시스템 문제 상황
- **장애 증상**: ES 컨테이너 노드가 다소 늦게 구동되거나 일시 정지 상태에 놓였을 때, 의존 앱(FastAPI 메인 서버) 단도 시작 기동을 지연한 채 20초 대기하다 결국 `Connection reset by peer` 혹은 `ValueError` 코드와 함께 전 서버가 허무하게 동반 뻗어 종료되는 재앙 발생.
- **근본 원인**: FastAPI `lifespan` 기동 시점의 ES 인덱스 초기화 매핑 필수 셋업 작업이 ES 기본 타임아웃 룰(10초) 동안 응답 대기하며 블로킹되는데, 실패 시 에러 예외 핸들링 방어 부재로 인해 콜스택 에러가 메인 엔진까지 전파됨.

### 5.2 패치 해결 방안
- 파이썬 ES 클라이언트 커넥션 연결 시 `request_timeout=1.0`, `max_retries=0`, `retry_on_timeout=False` 옵션을 명시하여 연결 불가 시 지체 없이 1초 컷으로 **즉각 실패(Fast-Fail)** 하도록 가혹하게 처리하고, 전체 구문을 `try-except` 블록망으로 래핑 처리함. 따라서 앱 진입 메인 코어는 어떻게든 무조건 계속 구동되어 화면을 유지하되 폴백 채널 우회망으로 비상 동작을 보장토록 패치함.

## 6. 관제망 거짓 CRITICAL 오발령 (False Positive 탐지 테러)

### 6.1 시스템 문제 상황
- **장애 증상**: 로그 메시지에 "error" 단어가 섞여버린 무해한 정상 통신 이력 (`GET /api/logs/download?level=CRITICAL HTTP/1.1 200 OK` 등) 때문에, Slack 알림 봇 엔진이 이를 실제 서버 붕괴 장애로 착각하고 알림을 사방에 브로드캐스팅해버리는 스팸 폭동 유발.

### 6.2 패치 해결 방안
- 로그 전처리 파서에 정규표현식(Regex) 감별 필터링 체인망을 추가 도입하여, HTTP 200/300번대 응답 로직이 존재하면 그 어떠한 피바람 부는 위험 키워드가 있더라도 무조건 강제로 안전한 `INFO` 레벨 등급으로 오버라이드(Override) 적용하여 거짓 오분류 전파망을 완벽히 소거 방어함.

## 7. ES 스토리지 초기화 시 좀비 로그 무한 환생 (Reingestion Loop) 버그

### 7.1 시스템 문제 상황
- **장애 증상**: ES 인덱스 초기화 버튼을 눌러 청소했음에도 불구, 무지성 `LogCollector` 폴링 수집 서버가 호스트 Docker 컨테이너의 옛날 파일 로그를 다시 처음부터 싹싹 긁어 읽어와, 초기화해둔 ES DB망에 옛날 에러를 또다시 박제해버리는 구형 좀비 로그 부활 사태 현상 발현.

### 7.2 패치 해결 방안
- 초기화 UI 클릭 시점의 UTC 서버 타임스탬프를 백엔드 인메모리에 전역 록 변수 플래그로 락 걸어 보전함. 
- Docker 수집 엔진 파서가 로그 스니핑 픽업 시 타임스탬프를 점검, 이 전역 허들 타임컷 기준선보다 1초라도 옛날에 발생한 구형 로그는 100% 원천 차단(Drop 버리기) 연산 처리하는 시간축 차폐 아키텍처 방어벽을 추가 건설함.
